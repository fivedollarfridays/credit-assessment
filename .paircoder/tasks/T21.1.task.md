# Objective

Replace free-text `list[str]` negative items with typed `NegativeItem` objects. Improve dispute pathway accuracy by using structured type/amount/date/status fields instead of string prefix matching. Maintain full backward compatibility — existing string-based payloads still work.

# Implementation Plan

1. Create `NegativeItemType` enum in types.py: LATE_PAYMENT, CHARGE_OFF, COLLECTION, IDENTITY_THEFT, WRONG_BALANCE, OBSOLETE_ITEM, UNAUTHORIZED_INQUIRY, DOFD_ERROR
2. Create `NegativeItem` Pydantic model in types.py:
   - type: NegativeItemType (required)
   - description: str (required, max 200 chars)
   - creditor: str | None (optional)
   - amount: float | None (optional, >= 0)
   - date_reported: str | None (optional, ISO date)
   - date_of_first_delinquency: str | None (optional, ISO date)
   - status: str | None (optional, e.g. "open", "closed", "paid")
3. Update `CreditProfile.negative_items` to accept `list[str | NegativeItem]`
   - Add a model_validator that coerces plain strings into NegativeItem by inferring type from prefix (same logic as current dispute_pathway.py)
4. Update `SimpleCreditProfile.negative_items` — keep as `list[str]`, coerce in `to_credit_profile()`
5. Update `dispute_pathway.py` to use `NegativeItem.type` field instead of string pattern matching
   - Fallback to string prefix matching for any items that arrive as raw strings (backward compat)
6. Update `assessment.py` negative item counting to work with both types
7. Update existing tests to cover both string and structured payloads
