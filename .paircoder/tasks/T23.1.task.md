# Objective

Create the DisputeRecord ORM model and repository for tracking dispute lifecycle. Model the status workflow (DRAFT through RESOLVED/ESCALATED) with FCRA deadline tracking.

# Implementation Plan

1. Create `DisputeStatus` enum: DRAFT, SENT, IN_REVIEW, RESPONDED, RESOLVED, ESCALATED
2. Create `DisputeRecord` ORM model in models_db.py:
   - id: primary key
   - user_id: str (indexed, from auth identity)
   - org_id: str | None (tenant isolation)
   - bureau: str (equifax/experian/transunion)
   - negative_item_data: JSON (serialized NegativeItem)
   - letter_type: str | None (if letter was generated)
   - status: str (DisputeStatus value)
   - round: int (default 1, increments on follow-up)
   - sent_at: datetime | None (when dispute was sent)
   - deadline_at: datetime | None (FCRA response deadline, auto-calculated)
   - responded_at: datetime | None
   - resolution: str | None (outcome description)
   - created_at: datetime (auto)
   - updated_at: datetime (auto)
3. Create `DisputeRepository` in repo_disputes.py:
   - create(dispute_data) -> DisputeRecord
   - get(dispute_id, user_id) -> DisputeRecord | None
   - list_by_user(user_id, status_filter, limit, offset) -> list[DisputeRecord]
   - update_status(dispute_id, user_id, new_status) -> DisputeRecord
   - count_by_user(user_id) -> int
   - get_approaching_deadlines(days_ahead=7) -> list[DisputeRecord]
4. Status transition validation:
   - DRAFT -> SENT (sets sent_at, calculates deadline_at = sent_at + 30/45 days)
   - SENT -> IN_REVIEW
   - IN_REVIEW -> RESPONDED (sets responded_at)
   - RESPONDED -> RESOLVED | ESCALATED
   - ESCALATED -> SENT (new round, round += 1, new deadline)
5. Alembic migration for dispute_records table
